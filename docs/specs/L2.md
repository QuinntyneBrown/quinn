# L2 Requirements — Quinn: Local LLM Coding Agent

---

## L1-1: CLI Interface

### L2-1.1: Entry Point and Argument Parsing
**Traces to:** L1-1
The CLI shall accept an optional inline prompt as a positional argument (e.g., `quinn "explain this code"`). It shall support flags: `--model` / `-m` to select an Ollama model, `--list-models` to show available models, `--system` / `-s` to provide a custom system prompt, and `--help` / `-h` for usage information. If no inline prompt is given, it enters interactive REPL mode.

**Acceptance Criteria:**
- Running `quinn --help` prints usage information and exits with code 0
- Running `quinn --list-models` prints available Ollama models and exits
- Running `quinn -m gemma3:4b "hello"` sends "hello" to gemma3:4b and prints the response
- Running `quinn` with no arguments enters interactive REPL mode

### L2-1.2: Interactive REPL
**Traces to:** L1-1
In interactive mode, the system shall display a prompt (e.g., `quinn> `), accept multiline input, and stream the LLM response to the terminal. The user can exit with `Ctrl+C`, `Ctrl+D`, or typing `/exit`. The REPL maintains conversation history within the session. Special commands: `/clear` resets conversation, `/model <name>` switches model, `/tools` lists available tools, `/help` shows commands.

**Acceptance Criteria:**
- The REPL displays a styled prompt and accepts input
- Typing `/exit` or pressing Ctrl+D exits cleanly
- `/clear` resets the conversation context
- `/model gemma3:4b` switches the active model
- `/tools` lists all registered tools with descriptions
- Conversation context persists across turns within a session

### L2-1.3: Streaming and Styled Output
**Traces to:** L1-1
LLM responses shall stream token-by-token to the terminal. The output shall use colored/styled text: tool calls are visually distinct from prose, code blocks are syntax-highlighted where feasible, and errors are shown in red. Markdown in responses is rendered appropriately for the terminal.

**Acceptance Criteria:**
- Tokens appear incrementally as they are generated (not all at once)
- Code blocks in responses are visually distinct (dimmed background or bordered)
- Error messages appear in red
- Tool invocations are displayed with a distinct prefix/style before execution

## L1-2: Local LLM Integration

### L2-2.1: Ollama HTTP Client
**Traces to:** L1-2
The system shall communicate with Ollama via its REST API (`http://localhost:11434`). The client supports the `/api/chat` endpoint with streaming, the `/api/tags` endpoint for listing models, and sends tool definitions using Ollama's tool calling format. The base URL is configurable via `OLLAMA_HOST` environment variable.

**Acceptance Criteria:**
- The client can call `/api/tags` and return a list of model names
- The client can send a chat request to `/api/chat` with streaming enabled and yield tokens
- The client respects `OLLAMA_HOST` env var for the base URL
- Connection errors produce a clear message ("Cannot connect to Ollama — is it running?")

### L2-2.2: Tool Calling Protocol
**Traces to:** L1-2
The system shall send tool definitions to the LLM using Ollama's native tool calling format in the chat API. When the LLM responds with tool calls, the system parses them. For models that do not support native tool calling, the system shall fall back to a prompt-based approach where tools are described in the system prompt and the LLM outputs a structured JSON block (```tool_call ... ```) which the system parses.

**Acceptance Criteria:**
- Tool definitions are included in the Ollama chat request when tools are available
- Native tool call responses from the LLM are correctly parsed into tool name + arguments
- The fallback prompt-based parser extracts tool calls from JSON code blocks
- Invalid or malformed tool calls produce an error message fed back to the LLM

### L2-2.3: System Prompt Construction
**Traces to:** L1-2
The system shall construct a system prompt that describes the agent's capabilities, lists available tools with their schemas, and instructs the LLM on how to use them. The system prompt includes the current working directory, platform info, and date. Users can append to the system prompt via the `--system` flag.

**Acceptance Criteria:**
- The system prompt includes the current working directory
- The system prompt includes descriptions of all registered tools
- The `--system` flag appends custom text to the system prompt
- The system prompt instructs the LLM on the tool call format

## L1-3: Tool System

### L2-3.1: Tool Registry and Interface
**Traces to:** L1-3
Each tool implements a common interface: `name`, `description`, `parameters` (JSON Schema), and an async `execute(args)` method returning a string result. Tools are registered in a central registry. The registry provides methods to list tools, get tool schemas (for LLM consumption), and execute a tool by name.

**Acceptance Criteria:**
- Tools implement name, description, parameters schema, and execute method
- The registry can list all tools with their descriptions
- The registry can return JSON-schema-formatted tool definitions for the LLM
- Calling `registry.execute("toolName", args)` invokes the correct tool

### L2-3.2: ReadFile Tool
**Traces to:** L1-3
Reads the contents of a file given an absolute or relative path. Supports optional `offset` (start line) and `limit` (number of lines) parameters. Returns the file contents with line numbers prefixed.

**Acceptance Criteria:**
- Reading an existing file returns its contents with line numbers
- The `offset` and `limit` parameters correctly slice the output
- Reading a nonexistent file returns a clear error message
- Binary files return a message indicating the file is binary

### L2-3.3: WriteFile Tool
**Traces to:** L1-3
Writes content to a file, creating it and any necessary parent directories. Overwrites the file if it exists.

**Acceptance Criteria:**
- Writing to a new path creates the file and parent directories
- Writing to an existing file overwrites it
- The tool returns confirmation with the file path and byte count

### L2-3.4: EditFile Tool
**Traces to:** L1-3
Performs a string replacement in a file. Takes `file_path`, `old_string`, and `new_string`. The `old_string` must match exactly one location in the file; if it matches zero or more than one, the tool returns an error.

**Acceptance Criteria:**
- A unique match is replaced and the updated file is written
- Zero matches returns an error: "old_string not found in file"
- Multiple matches returns an error: "old_string matches multiple locations — provide more context"
- The tool returns a diff-like summary of the change

### L2-3.5: Shell Tool (Bash)
**Traces to:** L1-3
Executes a shell command and returns stdout/stderr. Commands run in the project's working directory. Has a configurable timeout (default 120 seconds). Returns the exit code along with output.

**Acceptance Criteria:**
- A simple command (`echo hello`) returns its stdout
- A failing command returns stderr and non-zero exit code
- Commands exceeding the timeout are killed and return a timeout error
- The working directory is the project root

### L2-3.6: Glob Tool
**Traces to:** L1-3
Searches for files matching a glob pattern (e.g., `**/*.ts`) relative to the working directory. Returns matching file paths sorted by modification time.

**Acceptance Criteria:**
- A pattern like `**/*.ts` returns all matching TypeScript files
- No matches returns an empty list with a message
- Results are returned as a newline-separated list of relative paths

### L2-3.7: Grep Tool
**Traces to:** L1-3
Searches file contents for a regex pattern. Supports optional `glob` filter for file types and `context` lines. Returns matching file paths and line content.

**Acceptance Criteria:**
- Searching for a pattern returns matching lines with file paths and line numbers
- The `glob` filter restricts search to matching files
- The `context` parameter includes surrounding lines
- No matches returns a "no matches found" message

### L2-3.8: WebFetch Tool
**Traces to:** L1-3
Fetches a URL and returns the response body as text. Supports GET requests. This tool intentionally makes a network request — the agent should note this to the user.

**Acceptance Criteria:**
- Fetching a valid URL returns the response body text
- Network errors return a clear error message
- The tool output includes a notice that a network request was made

### L2-3.9: Playwright Browser Tool
**Traces to:** L1-3
Launches a headless browser to navigate to a URL, optionally take a screenshot, or extract page content. Playwright is an optional dependency — if not installed, the tool returns instructions to install it. This tool makes network requests.

**Acceptance Criteria:**
- If Playwright is not installed, returns a message with install instructions
- Navigating to a URL returns the page title and text content
- The screenshot action saves an image to a specified path
- Network errors return a clear error message

## L1-4: Agent Loop

### L2-4.1: Conversation Manager
**Traces to:** L1-4
Maintains an ordered list of messages (system, user, assistant, tool) for the current session. Provides methods to add messages, get the full history, and clear/reset. The conversation is ephemeral — it exists only for the session duration.

**Acceptance Criteria:**
- Messages are stored in order with their roles
- The full history can be retrieved for sending to the LLM
- Clearing the conversation resets to only the system message
- Tool results are stored as tool-role messages linked to their call IDs

### L2-4.2: Agent Execution Loop
**Traces to:** L1-4
The core loop: (1) send conversation to LLM, (2) stream the response, (3) if the response contains tool calls, execute each tool, append results, and go to step 1, (4) if the response is pure text, display it and return control to the user. The loop has a maximum iteration limit (default 25) to prevent infinite loops.

**Acceptance Criteria:**
- A user message that triggers no tool calls returns the LLM's text response
- A user message that triggers tool calls executes the tools and feeds results back
- Multi-step tool chains (tool call → result → another tool call) work correctly
- The loop terminates after the max iteration limit with a message to the user
- Each tool execution is displayed to the user before running

### L2-4.3: Error Handling
**Traces to:** L1-4
Tool execution errors are caught and returned to the LLM as error messages (not thrown). LLM connection errors are reported to the user with recovery suggestions. Malformed LLM output triggers a retry with an error note appended to the conversation.

**Acceptance Criteria:**
- A tool that throws an error returns the error message to the LLM, not to the user
- An Ollama connection failure displays a clear message and does not crash
- Malformed tool call JSON is reported back to the LLM for self-correction

## L1-5: Privacy and Security

### L2-5.1: No Remote LLM Calls
**Traces to:** L1-5
The system shall only communicate with `localhost` (or user-configured `OLLAMA_HOST`) for LLM inference. No LLM API calls go to any external service. The codebase contains no remote LLM endpoint URLs.

**Acceptance Criteria:**
- The only HTTP calls for inference go to the Ollama host (localhost by default)
- A code review of the LLM client confirms no remote endpoints
- The system works fully offline (given Ollama is running with a downloaded model)

### L2-5.2: No Telemetry
**Traces to:** L1-5
The system sends no analytics, telemetry, crash reports, or usage data anywhere. There are no tracking dependencies.

**Acceptance Criteria:**
- `package.json` contains no analytics/telemetry libraries
- A code review confirms no outbound data collection
- No environment variable or config option enables telemetry

### L2-5.3: Network-Aware Tool Execution
**Traces to:** L1-5
Tools that make network requests (WebFetch, Playwright) clearly indicate this in their description and output. The agent's system prompt instructs it to inform the user before making network requests.

**Acceptance Criteria:**
- WebFetch and Playwright tool descriptions mention network access
- Tool output includes a notice when a network request is made
- The system prompt instructs the LLM to notify users before using network tools
